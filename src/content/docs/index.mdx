## What makes UQL different

A UQL query is a plain JSON object — the same data you can log, cache, send over HTTP, or hand to the browser, with full TypeScript inference at every depth. That one decision shapes everything else:

- **Queries are data, not method chains.** Build them dynamically, store them, diff them, or send them straight from the client. There's no DSL to learn and nothing to compile.
- **No codegen, no build step.** Entities are TypeScript classes, so your code *is* the schema. There's no `.prisma` file to regenerate and no generated client to keep in sync.
- **One API everywhere.** The same syntax runs on PostgreSQL, MySQL, MariaDB, SQLite, LibSQL, Neon, Cloudflare D1, MongoDB, and Bun's native SQL.
- **Fast by design.** A zero-allocation engine measured at [3.9M+ ops/s](/comparison#performance) — on average ~2.4× faster than the next ORM in our [open benchmark](https://github.com/rogerpadilla/ts-orm-benchmark).

For an operation-by-operation code comparison against Prisma, Drizzle, TypeORM, and MikroORM — including where UQL is *not* the right pick — see the [full ORM comparison](/comparison).

## Who is UQL for?

UQL fits best when you want modern TypeScript ergonomics without trading away performance or portability:

- **AI and RAG applications** — native vector search, distance projection, and JSON-native queries for semantic search and retrieval pipelines.
- **Multi-tenant SaaS** — one schema and query model across every supported database, with entity-first migrations.
- **High-concurrency APIs** — a zero-allocation engine, sticky connections, and first-class streaming keep latency predictable under load.

The goal is Prisma-level ergonomics at query-builder speed.
